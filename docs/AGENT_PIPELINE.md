# AGENT_PIPELINE — конвейер ИИ-агента поиска информации

Спецификация ядра продукта: как бот превращает вопрос пользователя в точный ответ
со ссылками, оставаясь в рамках себестоимости. Документ ведёт реализацию **Этапов
2–5** (см. [`07_ROADMAP.md`](07_ROADMAP.md)) и фиксирует дизайн-решения, чтобы они
не жили только в чате.

> Статус: дизайн (draft). Уточняется по мере реализации Этапов 3–4. Числовые
> пороги (капы, confidence) — конфигурируемые, значения ниже — стартовые.

---

## 1. Принципы

1. **Агент, но управляемый.** Модель сама решает, уточнять ли и сколько искать —
   но всегда внутри жёстких капов (шаги + токены). Адаптивность от модели, потолок
   от нас.
2. **Точность через грундинг.** Факты берутся из веб-поиска и подаются с ссылками,
   а не из «памяти» модели. Нет источника — честно говорим, что не уверены.
3. **Дёшево на служебку, дорого на финал.** Сбор/решения — `flash-lite`, финальный
   ответ + поиск — `flash`.
4. **Одна точка списания.** Юзер платит за доставленный ответ; уточнения, поиск,
   ошибки — бесплатны.
5. **Не молчим.** Во время поиска бот пишет, что ищет.

## 2. Модели и роли

| Слой | Модель | Роль |
|---|---|---|
| Служебный | `google/gemini-2.5-flash-lite` | сбор ситуации, решения «хватает?/off-topic», генерация уточнений, классификация |
| Основной | `google/gemini-2.5-flash` | tool-calling поиск + синтез финального ответа со ссылками |

Обе — через OpenRouter, OpenAI-совместимый `tools`-API (function calling).

## 3. Общая схема конвейера

```
Вопрос юзера
   │
   ▼
[COLLECTING]  служебная модель (flash-lite), ИИ-driven, кап MAX_COLLECT_STEPS
   │  enough / confidence≥порог / step≥кап / «Ответить сейчас»
   ▼
[SEARCHING]   основная модель (flash) + tool web_search, кап MAX_SEARCH_STEPS + токен-бюджет
   │  модель решила, что данных хватает, ИЛИ упор в кап → форс-финал
   ▼
[ANSWERING]   синтез ответа со ссылками  →  ★ СПИСАНИЕ 1 запроса ★
   │
   ▼
[FOLLOWUP]    [Задать ещё] · [Новый диалог] · [Главное меню]
```

Off-topic / отмена / таймаут / ошибка — выход **без** списания на любом шаге.

## 4. Состояние COLLECTING (Этап 4)

Одно состояние с внутренним циклом, которым рулит служебная модель. Цель — добрать
недостающее для точного поиска, минимумом вопросов.

### 4.1 Карточка ситуации (`case`) — слоты юр-домена

Накапливается от шага к шагу; модель видит заполненное и спрашивает только пробелы.

| Слот | Смысл | Пример |
|---|---|---|
| `problem_type` | тип проблемы | не платят зарплату · увольнение · штрафы · проверка работодателя · трудовой договор · отпуск/больничный · другое |
| `region` | регион (юрисдикция/практика) | Москва · Свердловская обл. |
| `employment` | форма занятости | официально · неофициально · ГПХ · самозанятый · неизвестно |
| `timeline` | когда произошло / сроки | «задержка 2 месяца», «уволили вчера» |
| `documents` | какие документы есть | трудовой договор · расчётный лист · нет |
| `goal` | чего хочет добиться | вернуть деньги · оспорить · понять права · пожаловаться |
| `details` | свободное описание | текст пользователя |

Не все слоты обязательны — модель сама оценивает достаточность (`problem_type` +
`details` обычно минимум; `region`/`timeline` важны для сроков и практики).

### 4.2 Контракт решения (JSON от flash-lite)

На каждом сообщении пользователя служебная модель возвращает строго:

```json
{
  "off_topic": false,
  "enough": false,
  "confidence": 0.45,
  "case": {
    "problem_type": "не платят зарплату",
    "region": null,
    "employment": "официально",
    "timeline": "задержка 2 месяца",
    "documents": null,
    "goal": null,
    "details": "работодатель тянет с выплатой"
  },
  "missing": ["region", "goal"],
  "next_question": "В каком регионе вы работаете?",
  "quick_replies": ["Москва", "Санкт-Петербург", "Другой регион"]
}
```

- `off_topic: true` → мягкий редирект «я по рабочим/юридическим вопросам», **без**
  поиска и списания.
- `enough: true` **или** `confidence ≥ COLLECT_CONFIDENCE` **или** `step ≥
  MAX_COLLECT_STEPS` **или** кнопка «Ответить сейчас» → выход в SEARCHING.
- иначе → отправляем `next_question` + `quick_replies`, `step += 1`, остаёмся в
  COLLECTING.

### 4.3 Escape-механизмы

- **Ранний выход**: полный первый вопрос → 0 уточнений.
- **«Ответить сейчас»**: юзер обрывает сбор → сразу SEARCHING по тому, что есть.
- **«Отмена»**: → Главное меню, без списания.
- **Таймаут брошенного диалога** (Этап 8): нет ответа N минут → сброс, без списания.

### 4.4 Псевдокод

```python
async def collecting(ctx, user_msg):
    ctx.case.history.append(user_msg)
    d = await flash_lite.decide(ctx.case)          # JSON-контракт 4.2

    if d.off_topic:
        return redirect_offtopic()                 # без списания

    ctx.case.update(d.case)
    ctx.step += 1

    if d.enough or d.confidence >= COLLECT_CONFIDENCE or ctx.step >= MAX_COLLECT_STEPS:
        return goto(SEARCHING, summary=build_summary(ctx.case))

    send(d.next_question, buttons=d.quick_replies + ["Ответить сейчас", "Отмена"])
    return stay(COLLECTING)
```

## 5. Состояние SEARCHING (Этап 3)

Основная модель (`flash`) с инструментом `web_search` в tool-calling петле. Модель
сама решает: искать / добрать с другим запросом / хватит.

### 5.1 Инструмент

```
web_search(query: string) -> список результатов (title, url, snippet)
```

Единый интерфейс поверх провайдеров (Tavily основной, Exa/Firecrawl запасные —
см. §7). Ключи берутся из `app_settings` (правятся в `/admin`, Этап 7), применяются
горячо.

### 5.2 Цикл с капами

```python
async def searching(ctx, summary):
    messages = [system_answer_prompt(), user(summary)]
    for step in range(MAX_SEARCH_STEPS):          # напр. 4
        await notify_user_searching(step)         # «ищу информацию…» / «уточняю…»
        resp = await flash.chat(messages, tools=[WEB_SEARCH], token_budget=BUDGET)
        if resp.tool_calls:
            for call in resp.tool_calls:
                results = await run_web_search(call.query)   # §7 fallback
                messages.append(tool_result(call, results))
            continue
        return goto(ANSWERING, draft=resp.content, sources=collect_sources(messages))
    # упор в кап → форс-финал по собранному
    return goto(ANSWERING, forced=True, sources=collect_sources(messages))
```

Ограничители (все конфигурируемые):
- `MAX_SEARCH_STEPS` — потолок итераций поиска (старт: **4**).
- `BUDGET` — лимит токенов на весь вопрос; при превышении форс-финал.
- Форс-финал всегда даёт ответ по уже собранному, а не обрывается ошибкой.

### 5.3 «Не молчим»

Перед каждым шагом поиска — короткий статус пользователю (редактируем одно
сообщение): «Ищу информацию…», «Проверяю источники…», «Формирую ответ…».

## 6. Состояние ANSWERING (Этап 2 вводит списание; Этап 3 добавляет источники)

- Синтез финального ответа на русском, HTML-форматирование (`<b>`, `<i>`, `<code>`).
- **Ссылки на источники** обязательны, когда ответ опирался на поиск.
- Дисклеймер-оговорка для чувствительных тем («справочно, не юр-консультация»;
  для «проверки работодателя» — «данные из открытых источников»).
- **★ Точка списания ★**: ровно здесь, при успешной выдаче, `balance -= price`
  (`price` = цена запроса из настроек, дефолт 1). Больше **нигде** не списываем.

## 7. Fallback поисковиков

`run_web_search(query)` пробует провайдеров по порядку, пока не получит результат:

```
Tavily (основной)
  └─ ошибка/пусто/нет ключа → Exa (запасной)
        └─ ошибка/пусто → Firecrawl (запасной)
              └─ всё упало → отдаём модели пустой результат
                              (она это учтёт и честно сообщит)
```

Каждое переключение логируется. Активный провайдер и ключи — из `app_settings`.

## 8. Капы и контроль себестоимости (сводно)

| Кап | Стартовое значение | Где | Зачем |
|---|---|---|---|
| `MAX_COLLECT_STEPS` | 2–3 | COLLECTING | не допросить юзера до бесконечности |
| `COLLECT_CONFIDENCE` | 0.75 | COLLECTING | ранний выход, когда картина ясна |
| `MAX_SEARCH_STEPS` | 4 | SEARCHING | ограничить число поисков |
| `BUDGET` (токены) | задать по факту | SEARCHING | потолок стоимости вопроса |

Себестоимость одного вопроса при этих капах ≈ $0.01–0.03 худший случай. Цена
пакета для клиента = средняя себестоимость вопроса (с учётом цепочки) + наценка.

## 9. Точка списания и правила биллинга

1. Списание — **только** на входе в успешный ANSWERING. Одна точка.
2. Нет ответа (ошибка, пустой поиск, отмена, таймаут, off-topic) → **не** списываем.
3. Весь поиск и все уточнения (сколько бы шагов) → входят в **один** списываемый
   запрос.
4. `balance = 0` до начала → ветка вопроса ведёт в оплату, ИИ не вызывается.

## 10. Обработка нештатных ситуаций

| Ситуация | Поведение | Списание |
|---|---|---|
| off-topic | редирект «я по рабочим/юр-вопросам» | нет |
| ошибка ИИ/сети | «не получилось, попробуйте ещё раз» | нет |
| поиск ничего не нашёл | ответ «не нашёл точных данных» + предложить уточнить | нет |
| отмена/таймаут | возврат в меню | нет |
| смена темы в COLLECTING | `topic_changed` → сброс `case`, сбор заново | нет |

## 11. Логирование (приоритет заказчика)

Пишем в Loguru + `user_events` (Этап 9):
- вход в COLLECTING/SEARCHING/ANSWERING;
- каждый уточняющий вопрос и ответ юзера;
- каждый запрос к поисковику, выбранный провайдер, число результатов;
- число шагов поиска, факт форс-финала, итоговые источники;
- факт и размер списания.

Цель — по логам восстановить весь путь вопроса и быстро локализовать баг.

## 12. Скелеты промптов (уточняются при реализации)

**Служебный (COLLECTING, flash-lite)** — system:
> Ты помощник по трудовым/бытовым юридическим вопросам для рабочих. По сообщению
> пользователя заполни карточку `case` и реши, хватает ли данных для точного
> поиска ответа. Верни СТРОГО JSON по схеме (§4.2). Уточняющих вопросов — минимум,
> только по-настоящему недостающее. Вопросы — простым языком для аудитории 35–50.

**Основной (SEARCHING/ANSWERING, flash)** — system:
> Ты юридический ассистент. Отвечай точно и по делу, опираясь на веб-поиск
> (инструмент `web_search`). Если данных не хватает — ищи. Приводи ссылки на
> источники. Не выдумывай законы и суммы. Пиши на русском, HTML-форматирование.
> Добавляй оговорку, что ответ справочный, не заменяет юр-консультацию.

## 13. Открытые вопросы

- Точные числа капов и `BUDGET` — откалибровать на реальных вопросах (Этап 3).
- Нужен ли отдельный «гейт»: нужен ли вообще поиск (часть вопросов `flash` знает
  без веба) — потенциально большая экономия, решить на Этапе 3.
- Кэш частых ответов (топ-вопросы) — кандидат в отдельный этап/оптимизацию.
