# Дорожная карта реализации — Юр-бот «URIST2026»

11 этапов (0–10) от пустого репозитория до прод-готовности. Каждый этап
заканчивается **демонстрируемым результатом** (что можно потрогать руками).

Ключевая идея: **MVP-first**. Этап 1 = тонкая, но живая вертикаль (юзер видит
бота и меню). Дальше слои ИИ, поиска, оплаты и админки наслаиваются без
перепиливания базы.

> **Где смотреть прогресс**: текущий статус всех этапов — в [`STATUS.md`](STATUS.md).
> Внутри каждого этапа — чекбоксы конкретных acceptance-пунктов. Закрытие этапа
> сопровождается git-тегом `stage-N-done`.

| # | Этап | Главный артефакт (что демонстрируем) | Зависит от |
|---|---|---|---|
| 0 | Каркас репозитория | `docker compose up` поднимает бота, `/start` отвечает заглушкой, схема `urist_bot` создана миграцией | — |
| 1 | Онбординг + меню + БД юзера | Новый юзер видит приветствие+дисклеймер, в БД запись с балансом N, меню без тупиков | 0 |
| 2 | Базовый ИИ-ответ + память + списание | Вопрос → осмысленный ответ ИИ, баланс −1, память диалога работает | 1 |
| 3 | ИИ-агент с веб-поиском (Tavily) | Вопрос про свежие данные → «ищу…» → ответ со ссылками, в рамках капов | 2 |
| 4 | Адаптивный сбор ситуации (FSM) | Расплывчатый вопрос → бот уточняет (≤N) → отвечает; списание только за финал | 3 |
| 5 | «Проверить работодателя» | Ввод названия/ИНН → сводка из открытых источников + оговорка | 3 |
| 6 | Оплата ЮKassa (пакеты 10/20/30) | Покупка пакета в тесте → баланс +N, платёж в БД | 1 |
| 7 | Админка `/admin` | Админ меняет цену пакета/запроса и ключ поисковика → применяется без рестарта | 6 |
| 8 | Помощь + дисклеймеры + анти-абьюз | Экран помощи; абьюз упирается в лимиты | 2 |
| 9 | Аналитический слой БД | SQL показывает накопленные события (регистрации, запросы, платежи) | 2, 6 |

## Этап 2 — Базовый ИИ-ответ + память + списание

**Цель**: рабочее ядро — юзер задаёт вопрос и получает ответ ИИ, баланс списывается.

**Что делаем**:
- Клиент OpenRouter, модель `google/gemini-2.5-flash` для ответа (пока **без** веб-поиска).
- Ветка «Задать вопрос»: ввод → ответ ИИ (HTML) → списание 1 запроса → показ остатка.
- Память диалога в Redis (последние N сообщений), кнопка «Новый диалог» (сброс контекста).
- Баланс = 0 → ветка вопроса уводит в «Баланс и оплата» (заглушка оплаты).
- **Списание — одна точка**: только при успешной выдаче ответа.

**Критерии приёмки** (закрытие → tag `stage-2-done`):
- [x] Вопрос → осмысленный ответ ИИ на русском в HTML-форматировании. _(проверено: OpenRouter `gemini-2.5-flash`, ответ с `<b>`/списками; события «Ответ ИИ выдан» ×4)_
- [x] После ответа `balance −1`, юзеру показан остаток. _(проверено: баланс убывает; `charge_one` 2→1→0→None)_
- [x] Память работает: второй вопрос учитывает контекст первого; «Новый диалог» сбрасывает. _(проверено живьём: уточнение «А если откажут?» по контексту; «Новый диалог» ×2; память в Redis 4 msg)_
- [x] Ошибка ИИ / пустой ответ → баланс **не** списывается, юзер видит понятное сообщение. _(списание строго после успешного ответа; при AIError — текст без списания)_
- [x] При `balance = 0` вопрос не обрабатывается ИИ, юзер уводится в оплату. _(вход в ветку и повторная проверка перед вызовом уводят в экран оплаты)_

## Этап 3 — ИИ-агент с веб-поиском (Tavily + капы)

**Цель**: адаптивный поиск точной информации со ссылками, в рамках себестоимости.

> **Дизайн конвейера** — [`AGENT_PIPELINE.md`](AGENT_PIPELINE.md) (состояния SEARCHING/ANSWERING, цикл с капами, fallback поисковиков, точка списания).

**Что делаем**:
- Инструмент `web_search` (Tavily основной; Exa/Firecrawl — запасные через общий интерфейс).
- Tool-calling петля на `gemini-2.5-flash`: модель сама решает искать/добирать, **кап ≤3–4 шага + лимит токенов**, форс-финал при упоре в кап.
- «ИИ пишет, что ищет» — статус-сообщения во время поиска (не молчание).
- Ответ содержит ссылки на источники.

**Критерии приёмки** (закрытие → tag `stage-3-done`):
- [x] Вопрос, требующий свежих данных → бот показывает статус «ищу информацию…», делает поиск, отвечает со ссылками. _(живьём: «МРОТ 2026» → статус → ответ 27 093 ₽ + 5 источников)_
- [x] Петля не превышает кап шагов (проверяется по логам). _(лог «финал на шаге 1»; кап `max_search_steps=4`)_
- [x] Падение Tavily → автоматически используется запасной поисковик (Exa/Firecrawl). _(проверено: битый ключ Tavily 401 → Exa 5 рез.; все 3 ключа валидны)_
- [x] Весь поиск (сколько бы шагов) списывает с юзера **1** запрос. _(живьём: баланс 4→3 за один поиск)_
- [x] В логах видно: запросы к поисковику, число шагов, финальные источники. _(лог: «Поиск Tavily … → 5 рез.», «Агент: финал на шаге 1», «источников 5»)_

## Этап 4 — Адаптивный сбор ситуации (FSM + flash-lite)

**Цель**: бот сам добирает недостающие детали, но ограниченно и бесплатно для юзера.

> **Дизайн конвейера** — [`AGENT_PIPELINE.md`](AGENT_PIPELINE.md) (состояние COLLECTING, контракт решения JSON, слоты `case`, escape-механизмы).

**Что делаем**:
- Состояние `COLLECTING`: `gemini-2.5-flash-lite` возвращает structured JSON (enough/confidence/case-слоты/next_question/quick_replies/off_topic).
- Кап `MAX_STEPS` (≤2–3 уточнения), ранний выход по confidence, кнопка «Ответить сейчас».
- Off-topic → мягкий редирект без списания и без поиска.
- Компактная сводка `case` уходит дальше в поиск.

**Критерии приёмки** (закрытие → tag `stage-4-done`):
- [x] Расплывчатый вопрос → бот задаёт уточнение(я) с quick-replies, не больше `MAX_STEPS`. _(живьём: «не платят зарплату» → уточнение шаг 1/3, 4 варианта; кап `max_collect_steps=3`)_
- [x] Полный вопрос сразу → сбор пропускается (0 уточнений). _(живьём: «ООО НАШЕКРАШЕ» → «Сбор завершён шаг 1, enough → поиск»)_
- [x] «Ответить сейчас» обрывает сбор и уходит к ответу. _(живьём: «сломал станок» → уточнение → «Ответить сейчас» → финал, баланс 4→3)_
- [x] Уточнения **не** списывают запросы; списание — только за финальный ответ. _(живьём: балансы 5→4, 4→3 — по одному списанию за весь поток сбор+поиск)_
- [x] Не-юридический ввод → редирект без списания. _(живьём: «какая погода завтра» → off-topic редирект, баланс не изменился)_

- Статусы: оплачено (зачислить) / отклонено (повтор) / отмена-таймаут (в меню).
- Таблица `payments`, история покупок в «Баланс и оплата».
- Тестовые ключи ЮKassa из club_bot `.env`.

**Критерии приёмки** (закрытие → tag `stage-6-done`):
- [x] Покупка пакета 10 в тестовом режиме → `balance += 10`, запись в `payments`. _(живьём: buy:10 → «Пакет 10 зачислен, баланс 10»; payments id=3 succeeded/credited=t)_

## Этап 7 — Админка `/admin`

**Цель**: владелец управляет ценами и ключами поиска прямо из бота.

**Что делаем**:
- Доступ только для `admin_id`.
- Шапка: всего юзеров / платящих / неплатящих.
- Менять цены пакетов 10/20/30; менять цену 1 запроса (дефолт 1); менять API-ключи поисковиков (Tavily/Exa/Firecrawl).
- Значения в `app_settings` (БД), применяются **горячо** (без рестарта).
- **Балансы сервисов в шапке** (пожелание заказчика): остаток кредитов OpenRouter + расход/остаток по трём веб-поисковикам (Tavily/Exa/Firecrawl). Реализация: где у провайдера есть usage/billing-API — тянем реальный остаток (OpenRouter точно есть `GET /credits`; Firecrawl — вероятно; Tavily/Exa — проверить доки перед реализацией через EXA/Context7); где API нет — показываем **собственный счётчик израсходованных запросов** по провайдеру (данные уже копятся: каждый вызов поисковика логируется, добавим инкремент в БД).

**Критерии приёмки** (закрытие → tag `stage-7-done`):
- [x] `/admin` доступен только админу, чужим — отказ.
- [x] Шапка показывает корректные числа (сходятся с прямым SQL).
- [x] Смена цены пакета → новая цена в боте без рестарта.
- [x] Смена ключа поисковика → следующий поиск использует новый ключ.
- [x] Навигация админки без тупиков.
- [x] Шапка показывает баланс кредитов OpenRouter и расход/остаток по Tavily/Exa/Firecrawl (реальный остаток где есть API, иначе собственный счётчик расхода).
